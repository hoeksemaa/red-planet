# RP-12: Research + acquire high-quality Mars surface imagery tiles

## Decision

**Use the Casey Handmer / Murray Lab tiles already present in `data/raw/terraformed/`.**

The R2 bucket and local data are the same tileset. Zoom levels 0–8 are fully present on disk. No additional downloads needed. OPM/USGS tiles are a viable fallback but visually inferior and Cesium integration is more complex.

## What's Available Locally

`data/raw/terraformed/` is a complete gdal2tiles output tree:

- **Format:** TMS geodetic profile (EPSG:4326), 256×256 PNG tiles
- **Tile URL pattern:** `{zoom}/{x}/{y}.png` — same as OGC TMS spec
- **Zoom levels present:** 0 through 8 (confirmed by file presence of `0/0/0.png` through `8/0/0.png`)
- **Zoom levels declared:** tilemapresource.xml lists 0–8 (9 levels)
- **Tile count grows as expected:** zoom 0 = 2 tiles (2 columns, 1 row for geodetic), zoom 6 = 128 columns × 64 rows = 8192 tiles, etc.
- **Source VRT name:** `terraformed_7621_aa_dry.vrt` — Casey Handmer's terraformed-Mars color shaded relief at ~7m base resolution (from Murray Lab CTX mosaic)
- **Bounding box:** full planet (-180 to 179.996, -89.998 to 90.0)

The tiles are color shaded relief (not raw grayscale) — they already have the Mars rust/ochre palette with hillshading baked in. This is visually the best option by a wide margin.

## OPM / USGS Options (Evaluated, Not Recommended)

### OPM Mars Basemap v0.2 (Viking MDIM 2.1)
- **URL pattern:** `https://trek.nasa.gov/tiles/Mars/EQ/Mars_Viking_MDIM21_ClrMosaic_global_232m/1.0.0/default/default028mm/{z}/{y}/{x}.jpg`
  - Note: WMTS format — y/x swapped vs TMS standard; also requires WMTS provider, not UrlTemplate
- **Cesium provider:** `WebMapTileServiceImageryProvider`
- **Quality:** Low — Viking imagery from 1970s/80s, ~232m/px. Washed out, lower contrast, not color-accurate.
- **Pros:** Proven public access, no auth
- **Cons:** Visually inferior, worse LOD (fewer zoom levels), requires WMTS config

### USGS WMS (planetarymaps.usgs.gov)
- **URL:** `https://planetarymaps.usgs.gov/cgi-bin/mapserv?map=/maps/mars/mars_simp_cyl.map`
- **Cesium provider:** `WebMapServiceImageryProvider`
- **Quality:** Similar to Viking MDIM — acceptable but not great
- **Cons:** WMS is slower (server renders each request), no tile caching, no LOD, not suitable for smooth Cesium globe interaction

### Murray Lab CTX raw GeoTIFFs
- **~5.6 TB raw data** — not feasible without a tile pipeline and CDN. Do not download.

## Tile Scheme Deep-Dive: TMS Geodetic vs Cesium's Default

Critical gotcha: Cesium's `TileMapServiceImageryProvider` defaults to **WebMercator** (EPSG:3857). The Handmer tiles use **geodetic/geographic** (EPSG:4326). This must be specified explicitly.

In Cesium, geodetic TMS = `GeographicTilingScheme`. The geodetic scheme also uses a **2-tile root** (two 256×256 tiles cover the full planet at zoom 0: one for the western hemisphere, one for the eastern). The local tiles match this — zoom 0 has tiles `0/0/0.png` and `0/1/0.png`.

## Recommended Cesium Provider Config

### Option A — Local tiles (dev/offline, Vite static serving)

Vite serves `data/` as static assets from the `/data/` path prefix. The tiles are at `data/raw/terraformed/{z}/{x}/{y}.png`.

```typescript
const imageryProvider = await Cesium.TileMapServiceImageryProvider.fromUrl(
  '/data/raw/terraformed/',
  {
    fileExtension: 'png',
    tilingScheme: new Cesium.GeographicTilingScheme(),
    maximumLevel: 8,
  }
);
viewer.imageryLayers.addImageryProvider(imageryProvider);
```

`TileMapServiceImageryProvider.fromUrl()` reads `tilemapresource.xml` automatically if present — so specifying `tilingScheme` and `maximumLevel` here is belt-and-suspenders (the XML has the same info).

### Option B — R2 CDN (production, if access verified)

```typescript
const imageryProvider = await Cesium.TileMapServiceImageryProvider.fromUrl(
  'https://pub-3c6ee3900f804513bd3b2a3e4df337bd.r2.dev/',
  {
    fileExtension: 'png',
    tilingScheme: new Cesium.GeographicTilingScheme(),
    maximumLevel: 8,
  }
);
viewer.imageryLayers.addImageryProvider(imageryProvider);
```

If the R2 bucket also serves a `tilemapresource.xml`, Cesium will auto-configure from it.

### Option C — UrlTemplateImageryProvider (explicit, no XML dependency)

If the R2 bucket doesn't serve `tilemapresource.xml` or there are path issues:

```typescript
const imageryProvider = new Cesium.UrlTemplateImageryProvider({
  url: 'https://pub-3c6ee3900f804513bd3b2a3e4df337bd.r2.dev/{z}/{x}/{y}.png',
  tilingScheme: new Cesium.GeographicTilingScheme(),
  maximumLevel: 8,
  minimumLevel: 0,
  tileWidth: 256,
  tileHeight: 256,
});
viewer.imageryLayers.addImageryProvider(imageryProvider);
```

Note: `UrlTemplateImageryProvider` is synchronous (no `fromUrl` async call) — simpler to wire up.

## Implementation Recommendation

1. **Start with local tiles** (Option A / C with local path) — confirms visual quality with zero network dependency
2. **Switch to R2** (Option B / C with R2 URL) for production — same tiles, faster CDN delivery
3. **Fall back gracefully** — if R2 is unreachable, local tiles serve as dev baseline

The implementation agent (RP-7) should:
1. Wire up `TileMapServiceImageryProvider` pointing at `/data/raw/terraformed/` with `GeographicTilingScheme`
2. Add imagery as a named layer in the layer registry (RP-4 architecture)
3. Visually verify the tiles render correctly before moving to done
